# AI Tab Organizer V1 设计文档

## 1. 概述

AI Tab Organizer 是一个基于 Manifest V3 的 Chrome 扩展。它使用 Chrome
内置 Prompt API（Gemini Nano）在本地分析当前窗口的未分组标签页，并将
标签页整理到最多 5 个标签组中。

核心目标：

- 标签标题和 URL 不离开用户设备。
- 点击扩展图标或快捷键后，以尽可能少的操作完成分组。
- AI 不可用、超时或输出无效时仍有确定性的域名降级行为。
- 不修改已分组标签和固定标签。

V1 最低支持 Chrome 138，不兼容旧版 `ai.assistant` API。

## 2. 已确认的产品边界

### 2.1 处理范围

每次任务只处理触发时所在窗口中的候选标签。候选标签必须同时满足：

- 有有效的 `tabId`。
- `groupId` 表示未分组。
- 未固定。
- 仍位于触发任务的窗口。

少于 2 个候选标签时不执行分组，并显示“无需整理”。

已有标签组不参与重新分类，但其名称可提供给模型，供新标签优先复用。
同名现有组优先复用，并保留原有颜色。

### 2.2 分组约束

- 整次任务最多使用 5 个标签组，包括复用的现有组和新建组。
- 每个有效组至少包含 2 个本次候选标签。
- 零散或无法可靠归类的标签保持未分组。
- 中文、日文组名最多 4 个 Unicode 字符。
- 拉丁文字组名最多 12 个 Unicode 字符。
- 新建组从 Chrome 官方支持的颜色中分配颜色。

V1 不根据语义选择颜色。

### 2.3 中文支持

Prompt API 当前未正式承诺中文支持。V1 将中文能力定义为实验性能力：

- System Prompt 使用英文。
- 中文标题和清洗后的路径原样输入。
- 存在中文页面时要求模型输出中文组名。
- 不把中文声明为 API 的正式 expected language。
- 调用失败、输出无效或结果不可用时执行域名降级。

产品界面不得把中文分类描述为稳定或受 Chrome 官方保证的能力。

## 3. 技术方案

采用 TypeScript、Vite 和原生 Side Panel UI。

不使用 React。V1 的界面状态有限，引入框架不会带来足够收益。

### 3.1 Side Panel

Side Panel 是唯一调用 `LanguageModel` 的运行上下文，负责：

- 查询模型可用状态。
- 首次使用时显示“准备本地 AI”按钮。
- 在用户点击按钮后调用 `LanguageModel.create()`，触发 Chrome 管理的模型下载。
- 展示 Chrome 提供的下载进度。
- 运行批量推理和超时控制。
- 校验和合并模型输出。
- 在所有成功、失败、取消和超时路径的 `finally` 中调用
  `session.destroy()`。

模型已经可用时，Side Panel 打开后自动开始任务。

首次模型尚未下载时，不依赖打开 Side Panel 的用户激活传递。用户必须在
Side Panel 内点击一次“准备本地 AI”，下载完成后自动继续本次任务。

### 3.2 MV3 Service Worker

Service Worker 不调用 Prompt API，负责：

- 响应 Action 和 `Alt+Shift+G` 快捷键。
- 打开 Side Panel。
- 为每个窗口维护一个任务锁。
- 查询标签页和现有标签组。
- 执行 `chrome.tabs.group()` 和 `chrome.tabGroups.update()`。
- 向 Side Panel 提供可恢复的短期任务状态。
- 汇总成功、跳过和失败数量。

同一窗口只允许一个活动任务。重复触发只聚焦 Side Panel 并展示现有任务。
不同窗口可以分别拥有任务。

任务锁必须有过期时间。Service Worker 重启后发现过期锁时应自动清除，
避免永久阻塞。

### 3.3 纯逻辑模块

核心逻辑与 Chrome API 和 DOM 解耦，至少划分为：

- 候选标签过滤和快照。
- URL 规范化。
- 模型别名映射。
- 主域名解析。
- 首批轮询抽样和批次规划。
- Prompt 和结构化输出约束构建。
- 模型结果校验与跨批合并。
- 域名降级。
- 组名规范化。

这些模块使用 Vitest 进行单元测试。

## 4. 输入数据与隐私

### 4.1 URL 清洗

提供给模型的 URL 信息仅包含：

- 小写规范化后的 hostname。
- 最多前两级有效路径。

必须删除：

- 查询参数。
- URL hash。
- 用户名和密码。
- Token、会话标识和其他参数。

对于 `chrome://`、扩展页、新标签页等内部页面，只提供标题和页面类型，
不提供完整 URL。

### 4.2 标签别名

真实 `tabId` 不进入模型上下文。每批输入使用 `T1`、`T2` 等短别名，
扩展内部维护别名到真实 `tabId` 的映射。

已有标签组使用 `G1`、`G2` 等短别名。模型只接收组别名和名称，不接收
真实 group ID。

### 4.3 数据保留

扩展不得：

- 申请或发起任何第三方云端 AI 请求。
- 持久化标签标题、清洗后的 URL、Prompt 或模型原始输出。
- 在日志中记录上述内容。

任务结束后清除内存快照。日志只记录任务阶段、数量、耗时和稳定错误码。

## 5. AI 分类流程

### 5.1 建立快照

Service Worker 查询当前窗口并过滤候选标签。Side Panel 为模型准备清洗后的
临时快照。

任务执行前必须重新查询当前窗口。仅处理仍然存在、未固定、未分组且仍在
原窗口的标签。已关闭或状态已变化的标签直接跳过，不使整次任务失败。

### 5.2 分批策略

每批最多包含 50 个标签。

首批不按标签栏顺序或最近访问时间截取，而是按可注册主域名轮询抽样，
尽量覆盖更多网站和主题。主域名解析使用离线 Public Suffix List 数据，
不能通过简单截取 hostname 的最后两段实现。

首批负责建立本次任务的分类体系：

- 优先复用现有组。
- 可以创建新组。
- 已有组和新组总计最多 5 个。

后续批次只能把标签归入首批确定的分类，不能创建新分类。不匹配的标签
保持未分组。

### 5.3 Prompt 和结构化输出

使用 `LanguageModel` 和 `responseConstraint` 请求结构化 JSON。

模型返回值使用组别名或新组名称，以及标签短别名列表。Prompt 必须明确：

- 不输出 Markdown 或解释文字。
- 只能使用输入中存在的标签别名。
- 每个标签最多出现一次。
- 每组至少 2 个标签。
- 最多使用 5 个组。
- 后续批次只能使用既有分类。

结构化输出不能代替本地校验。

### 5.4 本地结果校验

执行前必须：

- 拒绝快照之外的标签别名。
- 去除重复标签，只保留第一次有效归属。
- 忽略不存在的组别名。
- 删除少于 2 个有效标签的组。
- 限制整次任务最多 5 个组。
- 去除组名首尾空白并按语言限制长度。
- 合并指向同一现有组或同一规范化名称的结果。

无效项可以单独忽略。若首批最终没有任何有效组，则整次任务切换到域名
降级。

## 6. 超时与部分结果

- 单批 AI 推理硬超时为 15 秒。
- 整次 AI 任务硬超时为 60 秒。
- 800ms 仅是模型已预热且输入较小时的性能目标，不是发布门槛。

首批调用失败、超时或无有效分类时，执行域名降级。

后续批次失败或总时间耗尽时：

- 停止后续 AI 调用。
- 保留此前已经校验的 AI 分类结果。
- 未处理或未匹配的标签保持未分组。
- 不把已经成功分类的标签重新改为域名分类。

## 7. 域名降级

以下情况自动触发域名降级：

- Prompt API 不存在。
- 设备或 Chrome 配置不支持模型。
- 模型下载失败。
- 模型创建失败。
- 首批推理失败、超时或无有效结果。

域名降级规则：

- 按可注册主域名聚类，例如 `docs.google.com` 和 `mail.google.com`
  归入 `google.com`。
- 只为至少有 2 个候选标签的域名创建组。
- 最多创建或复用 5 个组。
- 单标签域名和超出限制的域名保持未分组。
- Side Panel 明确显示本次使用了域名降级及稳定的原因类别。

域名降级不依赖模型，因此设备不支持 AI 时仍自动执行，不要求用户再次
点击。

## 8. Chrome 标签组执行

执行阶段由 Service Worker 逐组进行：

1. 重新校验标签当前状态。
2. 按规范化名称查找当前窗口中的同名组。
3. 存在同名组时，将标签加入该组并保留名称和颜色。
4. 不存在时创建新组，设置名称并分配 Chrome 支持的颜色。
5. 记录本组成功、跳过和失败数量。

采用尽力执行策略。某一组失败不回滚此前已成功的组，也不阻止后续独立组
继续执行。

## 9. Side Panel 状态

V1 至少支持以下状态：

- `idle`：等待任务或显示候选数量。
- `needs-setup`：首次使用，需要用户点击“准备本地 AI”。
- `downloading`：显示模型下载进度。
- `running`：显示当前批次、已处理数量和耗时。
- `completed`：显示 AI 分组结果摘要。
- `fallback-completed`：显示域名分组结果和降级原因。
- `no-op`：少于 2 个候选标签或没有可形成的有效组。
- `failed`：Chrome 分组 API 等无法通过域名算法恢复的错误。

Side Panel 重新加载时向 Service Worker 查询当前任务状态。完成或失败后
允许再次运行。

UI 视觉设计不属于本设计文档的定稿范围。

## 10. 权限与清单

Manifest V3 至少需要：

- `tabs`
- `tabGroups`
- `sidePanel`

是否需要额外权限必须在实现阶段以最小权限原则验证。清单中不得加入远程
AI 服务所需的 host permissions。

设置 `minimum_chrome_version` 为 `138`。低版本 Chrome 显示升级提示，
不承诺扩展核心流程可运行。

## 11. 测试与验收

### 11.1 单元测试

覆盖：

- URL 清洗和内部页面处理。
- 标签和标签组别名映射。
- Public Suffix List 主域名解析。
- 按主域名轮询抽样。
- 分批规划和分类体系冻结。
- 结构化结果校验、去重和跨批合并。
- 中日韩与拉丁组名限制。
- 域名降级和 5 组上限。

### 11.2 状态与集成测试

覆盖：

- 模型已可用时自动开始。
- 首次准备按钮和下载进度。
- 下载失败及自动域名降级。
- 单批 15 秒和总计 60 秒超时。
- 后续批失败时保留部分结果。
- Side Panel 重载恢复状态。
- 重复触发和窗口级任务锁。
- Service Worker 重启与锁过期。
- 复用同名组和创建新组。
- 标签中途关闭、移动、固定或手动分组。
- Chrome API 部分失败时不回滚。

### 11.3 人工验收

至少验证：

- 纯英文标签。
- 纯中文标签。
- 中英文混合标签。
- 超过 50 个候选标签。
- 单一域名大量标签和多域名混合标签。
- 当前窗口已有同名和不同名标签组。
- 模型不可用、输出无效和推理超时。

### 11.4 隐私验收

- 扩展无第三方 AI 网络请求。
- Manifest 不包含不必要的远程 host permissions。
- Chrome Storage 中不存在标题、URL、Prompt 和模型输出。
- 日志中不存在用户浏览内容。

## 12. 已知风险

- Chrome 对中文 Prompt API 的支持未正式稳定，分类质量和可用性可能随
  模型版本变化。
- 模型首次准备时间取决于网络、磁盘和设备条件，不能承诺 800ms。
- Side Panel 是稳定的模型运行载体，但不是最终理想交互。
- Public Suffix List 数据需要离线打包和定期更新，构建体积与准确性需要
  在实现阶段验证。
- MV3 Service Worker 会被挂起，任务状态设计不能依赖长期驻留内存。

## 13. 后续交互优化

理想交互是用户点击扩展图标后立即自动工作，不要求保持 Side Panel 可见。

后续版本探索使用标签栏中的临时进度组表达状态，例如“分析中”“整理中”
和进度变化。Chrome 标签组不能脱离真实标签存在，因此该体验只能通过
临时的扩展状态页标签和真实标签组模拟，而不是真正的虚拟组。

在采用该方案前必须验证：

- 临时标签和标签组对用户标签栏的扰动是否可接受。
- 是否会抢占焦点或改变当前活动标签。
- Side Panel 之外能否提供稳定的 Prompt API 文档上下文。
- 成功、失败、扩展重载和浏览器崩溃后的临时标签清理。
- 进度更新是否造成标签栏闪烁或顺序变化。
- 无障碍和快捷键用户能否理解当前状态。

该探索不进入 V1 实现范围，不应阻塞当前 Side Panel 方案。
